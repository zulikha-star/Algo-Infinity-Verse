1 · Array

Contiguous Block

All elements sit back-to-back in one unbroken block. Address of any element is computed directly — no traversal needed.

AccessO(1)
Extra overheadNone
Cache localityExcellent
Resize costO(n) — reallocate & copy

2 · Linked List

Scattered Nodes

Each node is its own tiny allocation, wherever the memory allocator finds space. Order only exists because each node points to the next one's address.

AccessO(n) — must traverse
Extra overhead+1 pointer per node (8 bytes, 64-bit)
Cache localityPoor
Insert/deleteO(1) once located

3 · Tree

Two Ways to Store the Same Shape

Trees can live in memory two very different ways: pointer-based nodes scattered like a linked list, or packed into a flat array using index math (common for heaps).

Pointer-based (scattered)

Each node stores its value plus left/right pointers (16 extra bytes on 64-bit). Nodes can end up anywhere in memory, unrelated to tree position.

Array-based (heap layout)

No pointers at all — a child's position is computed from its parent's index. Fully contiguous, but only practical for complete trees like heaps.

Access (pointer-based)O(log n) balanced
Overhead (pointer-based)2–3 pointers/node
Overhead (array-based)None — index math only
Cache localityPoor (pointer) / Excellent (array)

Try it: parent index → children (array-based)

Pick a parent index and compute — only indices 0–2 exist in this small example, but the formula works at any size.

4 · Hash Table

Contiguous Buckets, Scattered Collisions

The bucket array itself is one contiguous block — but when two keys hash to the same bucket, that bucket typically holds a small linked chain of scattered nodes.

Access (avg)O(1)
Access (worst)O(n) — all keys collide
Extra overheadEmpty slots (load factor < 1) + chain pointers
Cache localityGood for buckets, poor for chains

Try it: hash a key (capacity = 3)

Type a key and hash it to see which bucket it lands in — try a few different keys and watch for two landing in the same bucket, that's a collision.

5 · Queue

Contiguous, with Wraparound

Array-backed queues use two pointers — front and rear — and often wrap around the end of the buffer back to the start (a "circular buffer") instead of shifting everything on dequeue.

Try it: enqueue & dequeue

This is a 6-slot circular buffer — enqueue adds at REAR, dequeue removes from FRONT, and both wrap around to index 0 when they hit the end.

Enqueue/DequeueO(1)
Extra overheadTwo index pointers only
Cache localityExcellent
AlternativeLinked-list-backed = scattered, no resize needed
6 · Stack

Contiguous, Grows in One Direction

An array-backed stack is just a contiguous block with a single "stack pointer" tracking the top. Push/pop only ever touch that one end.

Try it: push & pop

Push and pop only ever touch the top — that's why both are O(1) regardless of how many elements are below.

Push/PopO(1)
Extra overheadOne stack-pointer index
Cache localityExcellent
Real-world exampleThe call stack itself works this way
Summary

All Six, Side by Side

The full picture in one table.

Structure Layout Extra memory / element Cache locality Typical access
Array Contiguous None Excellent O(1) by index
Linked List Scattered 1 pointer (8 bytes) Poor O(n) traversal
Tree (pointer-based) Scattered 2–3 pointers Poor O(log n) balanced
Hash Table Hybrid Empty slots + chain pointers Mixed O(1) average
Queue (array-backed) Contiguous 2 index pointers total Excellent O(1) enqueue/dequeue
Stack (array-backed) Contiguous 1 index pointer total Excellent O(1) push/pop

Why this matters: two structures with the same Big-O time complexity can perform very differently in practice because of cache locality. Contiguous structures let the CPU prefetch nearby memory efficiently; scattered structures cause cache misses on almost every step. This is part of why an array can outperform a linked list for sequential access even though both are "technically O(n)."
