Topic 1

The Call Stack

Every recursive call pushes a new frame onto the call stack. Every iterative loop just updates variables in place — no stack growth at all.

Recursive factorial(4)

function factorial(n) {
  if (n <= 1) return 1;       // base case
  return n * factorial(n-1);  // pushes a new frame
}

Each call waits on the stack for its child call to return before it can multiply and return its own result — 4 frames deep for factorial(4).

Iterative factorial(4)

function factorial(n) {
  let result = 1;
  for (let i = 2; i <= n; i++) {
    result *= i;               // updates in place
  }
  return result;
}

One stack frame for the whole function. result and i are just updated on every loop iteration — memory usage stays flat.

Topic 2

Tail Recursion

A tail-recursive function's last action is the recursive call itself — nothing left to do after it returns. That shape makes it rewrite-able into a loop.

Not Tail Recursive

function factorial(n) {
  if (n <= 1) return 1;
  return n * factorial(n-1);
  // ⬆ multiplication happens AFTER
  // the recursive call returns
}

Tail Recursive

function factorial(n, acc = 1) {
  if (n <= 1) return acc;
  return factorial(n - 1, n * acc);
  // ⬆ nothing left to do after this call —
  // the accumulator carries the work
}

Important: tail-call optimization (TCO) lets a compiler reuse the current stack frame instead of pushing a new one — but most JS engines and Python don't implement it. Languages like Scheme, Elixir, and some C compilers do. So in JS/Python, a tail-recursive function is still cleaner to reason about, but it won't save you from a stack overflow on very deep recursion — an explicit loop will.

Topic 5

Performance Comparison

Performance differences between recursion and iteration come down to two things: time complexity (does the algorithm redo work?) and space complexity (does the call stack grow?). Neither approach is "faster" by default — it depends entirely on whether the recursion re-solves overlapping subproblems.

Problem type Recursive time Iterative time Recursive space Iterative space
Linear pass (sum, search, reverse) O(n) O(n) O(n) — one stack frame per call O(1)
Divide & conquer (merge sort, binary search) O(n log n) / O(log n) same, with an explicit stack/queue O(log n) O(log n) with explicit stack, or O(1) for binary search
Naive overlapping subproblems (naive Fibonacci) O(2ⁿ) O(n) O(n) O(1)
Tree traversal (naturally recursive shape) O(n) O(n) with explicit stack/queue O(h) — tree height O(w) — widest level

The gap only becomes dramatic when recursion keeps re-solving the same subproblem — that's exactly what happens with naive Fibonacci, so it makes a good worked example of the worst case:

Recursive calls Iterative calls

Takeaway: this isn't "recursion is slow." It's that naive recursion without memoization re-does identical work exponentially. Add memoization (caching each fib(k) result) and the recursive version drops right back down to O(n) time — same complexity as the loop, just with O(n) stack space instead of O(1).

Topic 6

Common Mistakes

Click a card to reveal why each mistake breaks — and how to fix it.

Missing base case

Without a stopping condition, the function calls itself forever until the stack overflows. Fix: always define and test the base case first.

Off-by-one base case

Using n == 0 when the real stopping point is n == 1 (or vice versa) causes wrong results or one extra/missing call. Fix: trace the smallest 2–3 inputs by hand before trusting the recursion.

Stack overflow on large input

Deep recursion (e.g. processing 100,000 items) can exceed the call stack size. Fix: convert to an iterative loop or explicit stack for large inputs.

Recomputing overlapping subproblems

Naive recursive Fibonacci recalculates the same values thousands of times. Fix: add memoization, or switch to an iterative/tabulated approach.

Recursion where iteration is simpler

Summing a flat array recursively adds stack overhead for no real benefit. Fix: reach for recursion when the problem is naturally self-similar (trees, backtracking, divide & conquer) — not by default.

Mutating shared state across calls

Forgetting to pass a fresh copy of an accumulator/array leads to calls corrupting each other's data. Fix: pass new values as parameters rather than mutating outer-scope variables.

Exercises

Try solving each problem both ways, then reveal the solutions.

1. Sum of an array

Easy

Recursive

function sum(arr) {
  if (arr.length === 0) return 0;
  return arr[0] + sum(arr.slice(1));
}

Iterative

function sum(arr) {
  let total = 0;
  for (const n of arr) total += n;
  return total;
}

2. Reverse a string

Easy

Recursive

function reverse(s) {
  if (s === "") return "";
  return reverse(s.slice(1)) + s[0];
}

Iterative

function reverse(s) {
  let result = "";
  for (let i = s.length - 1; i >= 0; i--) {
    result += s[i];
  }
  return result;
}

3. Nth Fibonacci number

Medium

Recursive (naive)

function fib(n) {
  if (n <= 1) return n;
  return fib(n - 1) + fib(n - 2);
}

Iterative

function fib(n) {
  let [a, b] = [0, 1];
  for (let i = 0; i < n; i++) {
    [a, b] = [b, a + b];
  }
  return a;
}

4. Flatten a nested array

Medium

Recursive

function flatten(arr) {
  return arr.reduce((flat, item) =>
    flat.concat(Array.isArray(item) ? flatten(item) : item), []);
}

Iterative (explicit stack)

function flatten(arr) {
  const stack = [...arr];
  const result = [];
  while (stack.length) {
    const next = stack.shift();
    if (Array.isArray(next)) stack.unshift(...next);
    else result.push(next);
  }
  return result;
}

5. Binary tree height

Hard

Recursive

function height(node) {
  if (!node) return 0;
  return 1 + Math.max(height(node.left), height(node.right));
}

Iterative (BFS with explicit queue)

function height(root) {
  if (!root) return 0;
  let level = 0;
  let queue = [root];
  while (queue.length) {
    const next = [];
    for (const node of queue) {
      if (node.left) next.push(node.left);
      if (node.right) next.push(node.right);
    }
    queue = next;
    level++;
  }
  return level;
}

Trees are naturally recursive structures, so the recursive version here is usually the more readable choice in practice — this is a good example of when recursion is the better tool, not just the "convertible" one.
